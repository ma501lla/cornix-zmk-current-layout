# なぜ ZMK ファームウェアを使うのか

Cornix LP には標準で RMK ファームウェアが搭載されています。Vial アプリからリアルタイムにキーマップを変更でき、Bluetooth にも対応しています。それでも ZMK に切り替える意味はあるのでしょうか。

## 標準ファームウェア（RMK）と ZMK の比較

| | RMK（標準） | ZMK |
|---|---|---|
| Bluetooth 無線 | 対応（最大3台） | 対応（最大5台） |
| 左右間の無線通信 | 対応 | 対応 |
| キーマップ変更 | Vial GUI でリアルタイム | ファイル編集→ビルド→フラッシュ |
| 変更の手軽さ | とても簡単 | 慣れが必要 |
| Homerow mods | 限定的 | positional hold-tap で細かく制御可能 |
| コンボ | 基本的な機能 | レイヤー限定コンボ、タイムアウト個別設定など高度な制御 |
| Combo レイヤー | 非対応 | 対応（誤発動を根絶する設計が可能） |
| マクロ | 対応 | 対応 |
| 省電力 | 良好 | 良好 |
| 設定ファイル | `keyboard.toml`（TOML） | `.keymap`（DTS） |
| ビルド環境 | ローカルまたはクラウド | GitHub Actions（push で自動ビルド） |

## ZMK を選ぶ理由

### Vial では実現しにくいキーマップを組みたい

ZMK の最大のメリットは**キーマップのカスタマイズ自由度**です。

- **Positional Homerow mods**: ホームポジションのキーに修飾キーを割り当てつつ、左右どちらの手で押したかによって挙動を変えられる
- **Combo レイヤー**: 特定キーの長押しで専用レイヤーに移動し、そのレイヤー内だけで有効なコンボを定義できる。通常入力中のコンボ誤発動を根絶できる
- **レイヤー限定コンボ**: コンボの有効レイヤーを限定したり、キーごとにタイムアウトを調整できる
- **Sticky key コンボ**: 修飾キーをスタック可能な sticky key として使い、片手で Ctrl+Shift+キー のような操作ができる

### 設定をコードで管理したい

ZMK のキーマップはテキストファイル（DTS 形式）なので、Git でバージョン管理できます。変更履歴が残り、いつでも以前の設定に戻せます。GitHub にpush するだけで自動ビルドされ、ファームウェアがダウンロードできます。

### 接続台数が多い

ZMK は Bluetooth プロファイルを最大5つ持てるため、より多くのデバイスを切り替えて使えます（RMK は最大3台）。

## ZMK を選ばなくていい場合

- **Vial のキーマップで満足している** — 標準ファームウェアのままが最も手軽
- **リアルタイムに試行錯誤したい** — Vial の GUI でその場で変更できる体験は ZMK にはない（ZMK Studio で改善中）
- **トラブルを避けたい** — 標準ファームウェアは Cornix のサポート対象。ZMK は自己責任

## まとめ

| こんな人に | おすすめ |
|-----------|---------|
| キーマップにこだわりがない、手軽に使いたい | **RMK（標準）のまま** |
| Homerow mods やコンボを駆使したい | **ZMK を試す価値あり** |
| 設定を Git 管理して育てていきたい | **ZMK 向き** |
| Vial で十分、トラブルは避けたい | **RMK（標準）のまま** |
